1. Introduction

1.1 Document Purpose

The purpose of this document is to present a detailed description of READ. It will explain the goals of the project, the features implemented, the navigation of the user interface and finally the limitations and future-proofing aspects of this project.

1.2 Product Scope

The objective of the project is to create a web application to manage course materials for a class, track user’s face when reading study materials and provide important statistics based on it. There will be students’ and teachers’ portals, where the students can enroll in many courses and track their assignments and schedules all in one place. The teachers can share study materials and keep record of who actually read through everything using facial recognition and eye tracking and generate statistics for every individual class.

The main benefit associated with using this product is the unique feature which allows teachers to get back statistics on their students. Being able to evaluate the effectiveness of particular documents and understanding which type of study materials make students intrigued about a topic are among the myriad gaols of this product.

1.3 Intended Audience and Document Overview

This document is intended for the professor, project managers and developers who will find the list of functions and requirements needed for developing this application. This document does not illustrate any specific points of failure and uses some technical terms, so it may be less useful to testers and business managers. Section 3.2 of this document contains detailed description of the functionalities mentioned in section 2.2. Use case model is illustrated in section 3.3. The reader can jump to the relevant parts of the document as necessary.

1.4 Document Conventions

This document follows the IEEE formatting requirements. Each section and subsection have been divided and numbered to distinguish them. Special terms and comments are in italics.

2. Overall Description

2.1 Product Overview

The development of this product is primarily to be able to create a useful service for students and teachers, but also due to the necessity of a project for our course. This project does not follow on from any existing product, but is being built from scratch as a first iteration of a new product.

READ is designed to be an application for teachers and students, which is to be deployed as a website and as an android application. It acts as a powerful tool for teachers to maintain different classes of pupils and the course materials shared. READ allows teachers to check and verify (using facial recognition and eye tracking) who has read and completed their assigned tasks and aggregate statistics of class performance based on the records.

The front end will be a series of web pages and the back end will consist of a database system and a web server to run the program. Webcam will be accessed to track the users’ face and eyes.

2.2 Product Functionality

  1. Student and Teachers portals to track multiple courses
  2. Detect students’ faces when a reading material is viewed
  3. Identify if students’ eyes are open or close when reading material is viewed
  4. Record time spent on each study material
  5. Generate weekly reports of students’ activity/progress for teachers

2.3 Design and Implementation Constraints

  1. Manpower: limited to a group of four students
  2. Time: developers of this project have numerous other time consuming obligations/responsibilties
  3. Technical knowledge: our expertise is limited particularly with regard to the facial recognition / eye-tracking technology
  4. Availability of free APIs for facial recognition / eye-tracking

2.4 Assumptions and Dependencies

  1. Backend framework - Django
  2. Facial recognition API
  3. Eye tracking API
  4. Frontend libraries/frameworks for UI: (React, bootstrap)
  5. Android SDKs
  6. Federated ID provider: Google Sign-in
  7. Backend as a service: Firebase
  8. Server for deployment

3. Specific Requirements

3.1 External Interface Requirements

3.1.1 User Interfaces

The user interface will be several webpages each for a different task. The layout will be created using the help of any existing template.

  1. Registration and Login page - allows the user to register and create an account in the web application. If account already exists, the user will log in to their respective accounts.
  2. Dashboard page - the main page that will hold each user’s personal information and record of number of classes enrolled in. This includes a side menu bar that links to other pages.
  3. Classroom page - this page holds all the course materials or assignments of a specific class posted by a teacher and can be viewed by both students and teachers.
  4. Statistics page - this page can only be accessed by the teacher to see detailed information of class performance.

3.1.2 Hardware Interfaces

This product does not require any hardware interface because it is entirely enclosed within the bounds of a web application.

3.1.3 Software Interfaces

The majority of the web interface will be written in Html, Css and JavaScript, and so will not require any specialized software to run other than an up-to-date web browser. Django will be used as the framework to handle the backend which includes the database system and run the program. Additionally, a facial recognition API shall be used to meet the basic functionality of the product.

3.2 Functional Requirements

Functional requirements discussed in section 2.2 are elaborated below enlisting detailed requirements for every function.

3.2.1 Student and Teachers portals

There are two different options for user accounts - student’s account or teacher’s account. Each have different features available to them based on the type of account it is.

Requirements:
    Registration panel with two different options (student or teacher) requiring a username and a password.
    Login panel for users to login given they already have an account.

3.2.2 Facial Recognition and eye tracking

User will be prompted to allow webcam access so that their faces can be identified when reading a course material. Further more the users eye movements will be tracked to identify if they were actively reading or not. The time to read a document will also be recorded in the back end.

Requirements:
    every time a document is opened, activate webcam to detect face and eye movements and verify if the document has been read.
    record the result in a table

3.2.3 Generating useful statistics

For the teachers account, the teachers have the option to view statistics in the form of charts or graphs of students’ performance of each class. This includes average time to spent on reading course materials, record of which student viewed or didn’t view the assigned material, comparisons of different sections of the same class.

Requirements:
    button in teachers’ dashboard to statistics page
    show class performance for each course in a tabular form.

3.3.3 U1 – Create classroom

Actor - Teacher

Purpose – To create a classroom where students will be added and files can be uploaded

Priority - High

Preconditions – User must have an account

Post conditions – user will have a classroom created and a link to that page

Flow of Events

    1. User requests to create a new class
    2. User fills in information and creates a new class to add students in.

3.3.4 U2 – Upload files

Actor - Teacher

Purpose – upload files/documents for the students to read

Priority - High

Preconditions – User must have an account and a class created

Post conditions – user will upload any document in the classroom page

Flow of Events

    1. User requests to upload a file
    2. User selects the document from the options and clicks enter
    3. The new document is displayed in the classroom page

3.3.5 U3 – View statistics

Actor - Teacher

Purpose – teacher will be able to view statistics based on students’ performance

Priority - medium

Preconditions – User must have an account and a class created. User must have uploaded files

Post conditions – statistics page will show graphs and charts generated from students’ performance

Flow of Events

    1. User requests to view statistics page
    2. User selects different charts/graphs to view in details

3.3.6 U4 - enroll in class

Actor - Student

Purpose – student will join any class created by a teacher

Priority - high

Preconditions – User must have an account

Post conditions – User will be added to a classroom

Flow of Events

    1. User requests to join a class
    2. User is asked for a unique code
    3. Upon entering user can see all files uploaded by the teacher

3.3.7 U5 – View Documents

Actor – Student, Teacher

Purpose – teacher and students can view the documents once its uploaded

Priority - high

Preconditions – User must be enrolled in a classroom, there must be files uploaded

Post conditions – students’ face will be tracked and recorded

Flow of Events

    1. Students view a document
    2. Students face is tracked using facial recognition to check if they are reading
    3. The time to finish reading is recorded
    4. Teacher is notified about who has read through the given course material

4. Other Non-Functional Requirements

4.1 Performance Requirements

The website should be efficient on memory and loading times. Every page must load within a second.

4.2 Safety and Security Requirements

The website, once deployed on a web server rather than a local host, should get an HTTPS connection certificate, in order to ensure when a user enters the website, they know their data is secure and the connection is encrypted.

As this website will contain personal data regarding its users, security must be maintained. Users should be cautioned from putting sensitive data anywhere, and their passwords will be stored in hashed form in order to prevent brute-force access to their accounts. Users and Administrators should also be discouraged from sharing their account credentials, to prevent data breaches.

4.3 Software Quality Attributes

The main requirements here is that the system is straightforward and easy to use, therefore having good usability.

The main security concern here is user’s login information so a proper login mechanism must be used to avoid hacking.

5. Other Requirements

1.Federated User Credentials

In addition to making their own accounts whose credentials are stored in the project’s database, users will have the option of using their Google accounts to log in. As Google is one of the largest and widely used platforms on the internet. This would reduce the hassle for a new user of the website and improve ease of access. Again, security must be maintained on both the developers’ and users’ side, in order to ensure the users do not suffer any security breaches regarding their Google account.

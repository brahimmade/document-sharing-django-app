1 Introduction

This document provides a detailed description of the architecture of our product which is an online learning portal for teachers and students.

Section 2 lists the abbreviations used within the document.

All the components that will be used in building the product have been outlined in section 3. This includes the various technologies, programming languages, and tools used in the server side and the client side, as well as frameworks, libraries and APIs.

Section 4 contains a high level diagram of each of the main components which taken together will form the complete system. It shows the interactions between the different frameworks and subsystems, providing an overview of the main processes.

Class diagrams along with descriptions are given in section 5. These provide an overview of the different classes / entities of our system. This includes the attributes of each class and their corresponding methods. Inherited subclasses and specialisations are also shown.

Section 6 contains an ER diagram for the data model along with corresponding tables. This is consistent with the class diagrams, and shows the attributes of the different entities along with the relationship among them.

Section 7 contains a tentative implementation plan covering the order in which each feature is going to be built.

3 Description of Design Components

  • Server side
    1. Django framework: the python web framework that will be used to handle all the backend functionality. This framework handles a lot of the functionality that we would otherwise have to implement from scratch. This includes user authentication, content administration, security, commonly used templates, as well as a built in Object Relational Mapper (ORM).

    2. MySQL database: a popular SQL relational database that our team is familiar with. Rather than writing raw queries, which are error prone, we will use Django's ORM for all data interaction processes with a MySQL database.

    3. OpenCV (Facial recognition API): OpenCV is a real-time computer vision library providing a multitude of different features. It contains a facial recognition API with bindings through python, the programming language of choice for our backend functionality.

    4. Firebase: for some of the backend functionality within the android app of our product, we may require the use of services such as Firebase (e.g. some APIs are only available through Firebase).

  • Client Side
    1. HTML: the markup language for implementing the components viewable by the users of our product.

    2. CSS: will be used to style the components on the website, making it more appealing to the end user.

    3. Javascript: for manipulating the components of webpages, handling events, creating animations etc.

    4. jQuery (javascript library): allows HTML traversal, manipulation, event handling and other tasks to be done in a simpler fashion.

    5. Front end framework: we expect to be able to design a sufficiently captivating user-interface by using CSS and javascript. However, if the need arises, we may use a front-end framework such as React.

    6. Webgazer (Eye tracking API): this is an open source eye tracking API with a small footprint. Consequently, this can be directly embedded within the front-end functionality of our website using javascript.

6.2 Tables

Table descriptions

  1. Table: STUDENT
    • Purpose: Stores student information
    • Attributes
      1. s_id
        1. purpose: primary key of the table used to uniquely identify each student
        2. type: string of variable length
      2. name
        1. purpose: student's full name
        2. type: string of variable length
      3. email
        1. purpose: student's email address
        2. type: string of variable length
      4. password
        1. purpose: hashed password for this student's account
        2. type: string of length 128
      5. picture
        1. purpose: a photo of the student which will be used for facial recognition
        2. type: LONGBLOB
    • Foreign keys
      1. None
  2. Table: HAS_ACCESS_TO
    • Purpose: stores (student, document) pairs which will be used to generate statistics on how much of a document a student has read
    • Attributes
      1. s_id
        1. purpose: (student id) part of primary key of the table used to uniquely identify each (student, document) pair
        2. type: string of variable length
      2. d_id
        1. purpose: (document id) part of primary key of the table used to uniquely identify each (student, document) pair
        2. type: string of variable length
      3. time_spent
        1. purpose: total time a student has spend reading a document
        2. type: time
      4. pages_read
        1. purpose: number of pages of a document read by the student
        2. type: integer
    • Foreign keys
      1. s_id references STUDENT.s_id
      2. d_id references DOCUMENT.d_id
  3. Table: ENROLLED_IN
    • Purpose: stores (student_id, class_id) pairs to determine which students are enrolled into which classes
    • Attributes
      1. s_id
        1. purpose: (student id) part of primary key of the table used to uniquely identify each (student, class) pair
        2. type: string of variable length
      2. c_id
        1. purpose: (class id) part of primary key of the table used to uniquely identify each (student, class) pair
        2. type: string of variable length
    • Foreign keys
      1. s_id references STUDENT.s_id
      2. c_id references CLASS.c_id
  4. Table: CLASS
    • Purpose: stores information about each class
    • Attributes
      1. end_date
        1. purpose: stores the date when the class ends
        2. type: date
      2. start_date
        1. purpose: stores the date when the class starts
        2. type: date
      3. c_id
        1. purpose: stores the class_id which is the primary of this key
        2. type: string of variable length
      4. t_id
        1. purpose: stores the id of the teacher who teaches this class
        2. type: string of variable length
    • Foreign keys
      1. t_id references TEACHER.t_id
  5. Table: DOCUMENT
    • Purpose: stores information about a particular document
    • Attributes
      1. upload_date
        1. purpose: date on which the document was uploaded by the teacher
        2. type: date
      2. d_id
        1. purpose: (document_id) a unique identifier for each document used as the primary key
        2. type: string of variable length
      3. c_id
        1. purpose: the class_id for which this document is used
        2. type: string of variable length
    • Foreign keys
      1. c_id references CLASS.c_id
  6. Table: TEACHER
    • Purpose: stores information about each teacher
    • Attributes
      1. name
        1. purpose: stores teacher's full name
        2. type: string of variable length
      2. t_id
        1. purpose: (teacher_id) unique identifier for teacher, which will be used as the primary key
        2. type: string of variable length
      3. email
        1. purpose: stores teacher's email address
        2. type: string of variable length
      4. password
        1. purpose: stores password hash for a teacher account
        2. type: string of length 128
    • Foreign keys
      1. None

7 Implementation plan

  • Implementation of the website version
    1. Implement a minimial version of the UI.
    2. Implement the core backend functionality for the following features:
      • Teacher's login
      • Student's login
      • Classrooms
      • Teachers being able to send documents to students
      • Authenticating student users through facial recognition
      • Storing data on students' viewing of documents (including eye tracking data)
      • Generating statistics on each student based on stored data
    3. Extensively test the core functionality.
    4. Finish the UI design
    5. Implement any remaining backend functionality.
  • Implementation of the android app version
    1. Will be done similarly to the website version.
